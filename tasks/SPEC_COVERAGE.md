> ## Scope decision вЂ” v1 target, decided 2026-05-27
>
> The v1 target is the union of the modern-web baseline set and the formal W3C
> catalog set (every spec a current browser ships, plus the rest of the catalog
> at Candidate Recommendation). **Seven buckets are out of v1:** Ruby 1,
> Speech 1, Color HDR 6, Color Adjust 1, MathML Core, the Houdini Paint Worklet,
> and all print specs (Page 3, Paged Media, Generated Content for Paged Media).
> This is a deliberate deferral, not a permanent "never" вЂ” any of them can be
> re-evaluated after v1. Everything else is in v1 scope. The decision brought
> five specs back in that an older plan had deferred: Container Queries, Anchor
> Positioning, View Transitions, Scroll-Driven Animations, and the Typed object
> model. A рџљ« row below means one of the seven deferred buckets вЂ” nothing else.
> See `browser-plan/06_CSS.md` for the rule statement.

Default CI gates on `dotnet test --filter "Status!=Pending"` (skipped tests
don't fail the build). A separate non-gating CI job runs
`STARLING_RUN_PENDING=true dotnet test --filter "Status=Pending"` to spot
tests that have started passing вЂ” when one does, promote it from
`[PendingFact]` to `[SpecFact]` and update the row here.

> ## How spec coverage works вЂ” one source of truth
>
> **A spec test exists only if a human wrote it.** There is no
> auto-generation. Each conformance test is a normal test method that lives
> wherever it naturally belongs (the test project for the layer it
> exercises вЂ” `Starling.Layout.Tests`, `Starling.Paint.Tests`,
> `Starling.Css.Tests`, вЂ¦) and carries traits from `Starling.Spec.Common`:
>
> - `[Spec("<spec-id>", "<url>", section?)]` вЂ” what spec/section it covers.
> - `[SpecFact]` вЂ” asserts behavior we satisfy **today** (runs by default).
> - `[PendingFact("reason", trackingWp: "...")]` вЂ” a requirement we know
>   about but don't satisfy yet. Skipped by default; give it a real
>   assertion body so promotion is a one-line attribute flip.
>
> Filter by spec across *every* test project:
>
> ```bash
> dotnet test --filter "TestCategory~Spec:css-flexbox-1"   # one spec
> dotnet test --filter "Status!=Pending"                    # CI gate (passing only)
> STARLING_RUN_PENDING=true dotnet test --filter "Status=Pending"  # backlog / promote
> ```
>
> **`tasks/SPEC_CATALOG.md`** (generated by `SpecGen catalog` from the pinned
> `testdata/webref/css/*.json` snapshot вЂ” 123 specs, 1075 properties, 59
> at-rules, 169 selectors) is the *upstream map of what exists*. You find
> gaps by diffing the catalog against the `[Spec]`-tagged tests; the planned
> `SpecGen report` command (`wp:spec-tooling-bootstrap`) will automate that
> diff and rebuild the table below. Until then this file is hand-synced вЂ”
> last sync **2026-05-20**.
>
> `tests/Starling.Css.Spec.Tests/` still exists for CSS *parsing/cascade*
> conformance (it only references `Starling.Css`); behavioral layout/paint
> tests live in their own projects. Both surface under the same
> `TestCategory~Spec:<id>` filter.
